│   ├── 🍋Q01.为什么slice可以方便的进行扩容和传递？
│   └── 🍋Q02.为什么slice比数组使用广泛？
├── F02.答案📑
│   ├── S01.slice本身是一个结构体，维护着一个指向底层数组的指针，因此slice的大小总是固定的，因此方便传递

  1. slice本身是一个结构体，维护着一个指向底层数组的指针，因此slice的大小总是固定的，因此方便传递
  2. 可以通过内置append()方法对slice进行扩容，而数组类型不支持修改和扩容

├── 🤔Question(common)-005
│   ├── 🍋Q01.map查找元素的流程？
│   └── 🍋Q02.map定位key的流程？
├── F02.答案📑
│   ├── S01.根据key计算出**hash值**
│   ├── S02.取**hash值低位**和bucket数组的长度取模来确定bucket的位置
│   ├── S03.取**hash值高位**在**tophash数组**中查询
│   ├── S04.如果**tophash**中可以查询到，则定位到具体的key进行比较
│   ├── S05.如果**tophash**中查询不到，则去溢出桶中查，流程相同
│   ├── S06.如果溢出桶中也查不到，返回当前map的**value类型的零值**
│   └── S07.❗如果当前map正处于**搬迁**中，则会现在**老bucket数组**中查，查找不到再到**新bucket数组**中查
----------------------------------------------------------------------------------------

答案📑

  1. 根据key计算出hash值
  2. 取hash值低位和bucket数组的长度取模来确定bucket的位置
  3. 取hash值高位在tophash数组中查询
  4. 如果tophash中可以查询到，则定位到具体的key进行比较
  5. 如果tophash中查询不到，则去溢出桶中查，流程相同
  6. 如果溢出桶中也查不到，返回当前map的value类型的零值
  7. ❗如果当前map正处于搬迁中，则会现在老bucket数组中查，查找不到再到新bucket数组中查

├── F02.答案📑
│   ├── S01.string底层其实是一个结构体
│   ├── S02.包含一个执行底层`[]byte`的指针

  4. 另一方面，Go语言约定string为值类型，值类型作为形参和返回值都是拷贝，如果string的底层[]byte可修改，就会存在很多莫名奇妙的错误

问题背景📑

  1. 管道是Go在语言层面提供的协程间的通信方式
  2. 比UNIX管道更易用也更轻便

特征📑

阻塞的情况

  1. 向（满chan | nil chan）中写
  2. 从（空chan | nil chan）中读

panic的情况

  1. 向关闭的chan(管道)中写数据
  2. 关闭已经关闭了的chan(管道)

并发安全

用法📑

初始化

  1. 内置函数make()进行初始化，可创建无缓冲管道和有缓冲管道

查询缓冲区

  1. len(ch)：缓冲区中元素个数
  2. cap(ch)：缓冲区容量

普通读写

  1. vxx <- ch👉从管道中读取数据，存入vxx中
  2. ch <- xxx👉向管道中写数据xxx

❤️逗号模式读写

  1. val,ok <- ch👉使用逗号模式从管道中读取数据，ok为bool类型，表示是否读取到数据
  2. 推荐方式，管道关闭后读取管道需通过ok判断本次是否读取到数据，以防返回零值，误操作
  3. 管道关闭&管道缓冲区中无数据 ⇌ ok为false

单向管道

  1. 顾名思义，单向管道指只能发送或接收数据的管道类型
  2. 通过在函数参数或函数返回值中用管道操作符(<-)进行限定管道的读写属性

select

  1. select通常配合管道使用，可以监控多个管道的事件
  2. 可以看作是Go在语言层面的多路复用机制

for-range

  1. for-range用于从chan(管道)中持续读取数据
  2. 管道中无数据&管道未关闭 ⇌ for-range阻塞等待
  3. 管道中无数据&管道关闭 ⇌ for-range结束
  4. 管道中有数据&管道关闭 ⇌ for-range持续读取数据直至读取完，结束

原理📑

环形队列

  1. chan内部实现了一个环形队列作为其缓冲区
  2. 队列的容量在使用make()创建chan时指定
  3. chan内部有两个指针分别指向下一个要读取的元素的位置和下一个要写入的元素的位置

等待队列

  1. chan内部维护了两个协程等待队列：读阻塞协程队列和写阻塞协程队列
  2. 因读阻塞的协程队列会被向管道写入数据的协程唤醒
  3. 因写阻塞的协程队列会被从管道读取数据的协程唤醒

类型信息

  1. chan内部维护了元素的类型信息，因此再创建chan时一旦指定了元素类型，后续就只能传递这一种类型
  2. 如果需要chan(管道)传递任意类型数据，则可以使用interface{}类型

互斥锁

  1. chan中维护了一个互斥锁
  2. 因此chan(管道)同时仅允许被一个协程读写

问题背景📑

  1. slice又称动态数组，依托数组实现，可以方便的进行扩容和传递
  2. 实际使用中比数组更广泛和灵活

特征📑

并发不安全

panic的情况

  1. 索引超出范围

用法📑

初始化

  1. 内置函数make()进行初始化，可指定slice长度和容量（即底层数组的长度）
  2. 创建切片时根据实际需要预分配容量，尽量避免追加过程中扩容，有利于提升性能
  3. []type{}进行初始化

slice拷贝

  1. copy()方法用于将源slice中的数据拷贝到目标slice
  2. 拷贝的元素个数取源slice和`目标slice元素个数的最小值
  3. 拷贝的过程，从第一个元素开始，直到达到拷贝的元素个数
  4. 拷贝的过程中不会发生扩容

切片操作⇨简单表达式([low:high])

  1. 通过对数组或其它slice进行切片(arr[low:high] | s1[low:high])可以获得一个新的slices2
  2. 0<=low<=high<=len(arr)=cap(arr);0<=low<=high<=cap(s1)
  3. 新的slice会和数组或原slice共享底层数组
  4. 简单表达式切片并未限制新的slice的容量，也就是说通过append()添加新元素时有覆盖原数组或者原slice元素的风险
  5. 即arr或s1的[high:cap(arr)|high:cap(s1)]这一块可能被s2覆盖
  6. 是否会覆盖则要看通过append()添加新元素时是否会触发slice扩容，触发扩容就不会覆盖，否则，就会覆盖

切片操作⇨扩展表达式([low:high:max])

  1. arr[low:high:max] | s1[low:high:max]👉s2
  2. 0<=low<=high<=max<=len(arr)=cap(arr);0<=low<=high<=max<=cap(s1)
  3. 扩展表达式切片限制了新的slice的容量，相当于一个被封印了的切片，通过append()添加新元素时，如果存储容量不足，则会触发扩容，不会覆盖意料之外的数据

切片操作⇨简单表达式

  1. [⚪,🟡,⚪,⚪,🟢,⚪,⚪]
  2. low=🟡，high=🟢，虽然新slice的len从🟡到🟢，但cap却是从🟡到末尾，append()添加新元素时🟢之后的空间有被覆盖的风险

切片操作⇨扩展表达式

  1. [⚪,🟡,⚪,⚪,⚪,🟢,🟠,⚪,⚪]
  2. low=🟡，high=🟢，max=🟠，len从🟡到🟢，cap从🟡到🟠，append()添加新元素时🟠之后的空间没有被覆盖的风险
  3. 至于🟢到🟠的这一块是我们自己控制的，默认可覆盖，我们完全可以让🟢=🟠

原理📑

slice结构体

  1. data：元素存哪里
  2. len：存了多少个元素
  3. cap：可以存多少个元素

slice扩容规则

  1. 使用append向slice追加元素时，如果slice容量不足，则会触发slice扩容
  2. 扩容实际上是分配一块更大的内存，将原slice数据拷贝进新slice，然后返回新slice，扩容后再将数据追加进去
  3. Go一直在优化slice的扩容规则，这些优化体现在了两方面：①slice性能②slice空间使用率
  4. 优化的目的也是使slice性能和slice空间使用率这两方面都相对较好，提高slice的综合性能

slice扩容时有几个关键的变量

  1. oldCap：扩容前容量
  2. oldLen：扩容前元素个数
  3. cap：扩容所需最小容量
  4. newCap：预估容量

Go1.15扩容规则

  1. 如果oldCap(扩容前的容量)翻倍之后还是小于cap(扩容所需最小容量)，那么newCap(预估容量)就等于cap(扩容所需最小容量)
  2. 如果不满足第一条，而且oldLen(扩容前元素个数)小于1024，那么newCap(预估容量)=oldCap(扩容前的容量)*2
  3. 如果不满足第一条，而且oldLen(扩容前元素个数)大于等于1024，那就循环扩容四分之一，直到大于等于所需最小容量

Go1.16扩容规则

  1. Go1.16中有了些变化，和1024比较的不再是oldLen(扩容前元素个数)，而是oldCap(扩容前的容量)

Go1.18扩容规则

  1. 到了Go1.18时，又改成不和1024比较了，而是和256比较；
  2. 并且扩容的增量也有所变化，除了每次扩容1/4，还得加上256*3/4

预估容量不一定为最终申请的容量

  1. Go的内存管理，申请内存时都有一定的规格(8,16,32,48…)
  2. e.g. int类型slice，预估容量为5，64位操作系统，需要申请40字节，但内存规格中不存在40，Go的内存管理会帮我们匹配到足够大、且最接近的规格48，最终申请的容量是6

问题背景📑

  1. 字典类型
  2. 底层使用Hash表实现

特征📑

并发不安全

  1. 多协程同时同时写会报错concurrent map writes

panic的情况

  1. 给nil map添加key-value

用法📑

初始化

  1. 使用内置函数make()进行初始化，可以同时指定map的容量，根据需要指定容量可以有效减少内存分配次数，有利于提升性能

删除

  1. 使用内置函数delete()进行删除

逗号模式(val,ok)获取值

  1. 查询map时，使用逗号模式获取值，避免操作零值

原理📑

map结构体

  1. 元素个数
  2. bucket数组大小
  3. 指向bucket数组的指针
  4. 指向老bucket数组的指针(扩容时使用)

bucket结构体

  1. tophash：存储Hash值的高8位；一个bucket可以存储8个元素，Hash值低位相同的键会存到同一个bucket中，将这些键的高8位存储到tophash中，方便后续查询
  2. data：存储key,value；为了节省字节对齐带来的空间浪费，先存的是key，再存value(key/key/key···value/value/value···)
  3. overflow：存储溢出bucket的地址；当当前bucket存满了，使用拉链法解决冲突，新元素存到溢出bucket中

map负载因子

  1. 负载因子=键数量/bucket数量
  2. 负载因子小，说明空间利用率不高
  3. 负载因子大，说明冲突严重，存取效率低

扩容规则

  1. 负载因子大于6.5
  2. overflow(溢出桶)的数量达到一定值
  3. 👆满足上述任意条件触发扩容
  4. 扩容的过程其实是重新创建bucket，重新组织键值对，让键值对均匀分布到新创建的bucket中，这个过程称为rehash

增量扩容

  1. 负载因子大于6.5时，触发增量扩容
  2. 新建一个bucket数组，长度是之前bucket数组的2倍，将之前bucket的元素搬迁过来即可
  3. map结构体中维护了执行新bucket数组和老bucket数组的指针
  4. Go语言采用逐步搬迁的策略，每次访问map时都会触发一次搬迁，每次搬迁2个bucket，直至搬迁完毕，就可以释放老的bucket数组了

等量扩容

  1. 极端场景下，比如经过大量的元素增删时，键值对刚好集中在一小部分的bucket中，这样会造成溢出的bucket数量增多，触发扩容的第二个条件overflow(溢出桶)的数量达到一定值
  2. 此时会触发增量扩容，增量扩容并不该表map容量，但也会走一遍等量扩容的流程
  3. 目的在于重新组织键值对，让键值对均匀分布到新创建的bucket中

问题背景📑

  1. sync.Map是一种并发安全的map

特征📑

零值

  1. sync.Map是结构体，零值是结构体的零值，并非nil
  2. 因此声明完即可使用

遍历方法

  1. sync.Map不能使用for-range遍历
  2. sync.Map提供了range方法支持遍历，并且可以针对每个键值对设置回调方法

并发安全

  1. sync.Map并发安全，支持并发读写，因此遍历期间有可能读到其它协程刚写入的数据，类似于脏读

用法📑

增删改查

  1. 增改：sync.Map.Store(key,value)
  2. 查：sync.Map.Load(key)
  3. 删：sync.Map.Delete(key)
  4. 遍历：sync.Map.Range(f func(key, value any) bool)

LoadOrStore

  1. 如果key存在，则执行Load方法，加载原来的数据
  2. 如果key不存在，则执行Store方法，插入数据

LoadAndDelete

  1. 如果key存在，则执行Load和Delete方法，删除键值对并返回删除的值
  2. 如果key不存在，则返回标识key不存在

原理📑

sync.Map数据结构

  1. sync.Map内部维护了两个mapread和dirty，read用于并发读，dirty用于新数据的写入
  2. sync.Map内部维护了一个互斥锁用于保护dirty表
  3. dirty用于新数据的写入，访问需要加锁，但dirty表也只是暂存数据，最终都会同步到read表中
  4. sync.Map内部维护了一个整型值用于记录查找read表时miss的次数
  5. 上面说的dirty表暂存数据，最终都会同步到read表中；同步的条件就是miss的值累加到一定的值(等于数据总数)
  6. dirty表向map同步数据的过程中也会加锁，避免多个同步操作并发执行
  7. 因此，sync.Map在读多写少且键值对不断增长的场景下可以减少对锁的依赖，因为read表只读，可并发读写
  8. sync.Map缩小了互斥锁的保护范围，获得性能提升
