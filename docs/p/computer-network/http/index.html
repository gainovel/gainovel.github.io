问题背景📑

  1. HTTP 响应状态码用来表明特定 HTTP 请求是否成功完成。 响应被归为以下五大类
  2. 信息响应 (100–199)、成功响应 (200–299)、重定向消息 (300–399)、客户端错误响应 (400–499)、服务端错误响应 (500–599)

列举📑

1xx

  1. 👇1xx 类状态码属于提示信息，是协议处理中的一种中间状态，实际用到的比较少
  2. 100 Continue 这个临时响应表明，迄今为止的所有内容都是可行的，客户端应该继续请求，如果已经完成，则忽略它。

2xx

  1. 👇2xx 类状态码表示服务器成功处理了客户端的请求
  2. 「200 OK」是最常见的成功状态码，表示一切正常。如果是非 HEAD 请求，服务器返回的响应头都会有 body 数据
  3. 「204 No Content」也是常见的成功状态码，与 200 OK 基本相同，但响应头没有 body 数据
  4. 「206 Partial Content」是应用于 HTTP 分块下载或断点续传，表示响应返回的 body 数据并不是资源的全部，而是其中的一部分，也是服务器处理成功的状态。

3xx

  1. 3xx 类状态码表示客户端请求的资源发生了变动，需要客户端用新的 URL 重新发送请求获取资源，也就是重定向
  2. 「301 Moved Permanently」表示永久重定向，说明请求的资源已经不存在了，需改用新的 URL 再次访问
  3. 「302 Found」表示临时重定向，说明请求的资源还在，但暂时需要用另一个 URL 来访问
  4. 301 和 302 都会在响应头里使用字段 Location，指明后续要跳转的 URL，浏览器会自动重定向新的 URL
  5. 「304 Not Modified」不具有跳转的含义，表示资源未修改，重定向已存在的缓冲文件，也称缓存重定向，也就是告诉客户端可以继续使用缓存资源，用于缓存控制

4xx

  1. 4xx 类状态码表示客户端发送的报文有误，服务器无法处理，也就是错误码的含义
  2. 「400 Bad Request」表示客户端请求的报文有错误，但只是个笼统的错误
  3. 「403 Forbidden」表示服务器禁止访问资源，并不是客户端的请求出错
  4. 「404 Not Found」表示请求的资源在服务器上不存在或未找到，所以无法提供给客户端

5xx

  1. 5xx 类状态码表示客户端请求报文正确，但是服务器处理时内部发生了错误，属于服务器端的错误码。
  2. 「500 Internal Server Error」与 400 类型，是个笼统通用的错误码，服务器发生了什么错误，我们并不知道
  3. 「501 Not Implemented」表示客户端请求的功能还不支持，类似“即将开业，敬请期待”的意思
  4. 「502 Bad Gateway」通常是服务器作为网关或代理时返回的错误码，表示服务器自身工作正常，访问后端服务器发生了错误
  5. 「503 Service Unavailable」表示服务器当前很忙，暂时无法响应客户端，类似“网络服务正忙，请稍后重试”的意思

问题背景📑

  1. HTTP 标头（header）允许客户端和服务器通过 HTTP 请求（request）或者响应（response）传递附加信息
  2. 一个 HTTP 标头由它的名称（不区分大小写）后跟随一个冒号（:），冒号后跟随它具体的值。该值之前的空格会被忽略

列举📑

Host 字段

  1. 客户端发送请求时，用来指定服务器的域名
  2. 有了 Host 字段，就可以将请求发往「同一台」服务器上的不同网站
  3. xxx
  4. 👇xxx👇
  5. xxx
  6. 🚩xxx🚩

Content-Length 字段

  1. 服务器在返回数据时，会有 Content-Length 字段，表明本次回应的数据长度
  2. HTTP 是基于 TCP 传输协议进行通信的，而使用了 TCP 传输协议，就会存在一个“粘包”的问题
  3. HTTP 协议通过设置回车符、换行符作为 HTTP header 的边界，通过 Content-Length 字段作为 HTTP body 的边界，这两个方式都是为了解决“粘包”的问题

Connection 字段

  1. Connection 字段最常用于客户端要求服务器使用「HTTP 长连接」机制，以便其他请求复用
  2. HTTP 长连接的特点是，只要任意一端没有明确提出断开连接，则保持 TCP 连接状态
  3. HTTP/1.1 版本的默认连接都是长连接，但为了兼容老版本的 HTTP，需要指定 Connection 首部字段的值为 Keep-Alive

Content-Type 字段

  1. Content-Type 字段用于服务器回应时，告诉客户端，本次数据是什么格式
  2. Content-Type: text/html; Charset=utf-8 ⇨ 表明发送的是网页，而且编码是UTF-8

Accept 字段

  1. Accept 字段声明自己可以接受哪些数据格式
  2. Accept: / ⇨ 自己可以接受任何格式的数据

Content-Encoding 字段

  1. Content-Encoding 字段说明数据的压缩方法。表示服务器返回的数据使用了什么压缩格式
  2. Content-Encoding: gzip ⇨ 表示服务器返回的数据采用了 gzip 方式压缩，告知客户端需要用此方式解压

Accept-Encoding 字段

  1. Accept-Encoding 字段说明自己可以接受哪些压缩方法
